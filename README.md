# StudentClaw 真正的能工智人

[English](README.en.md)

> 微信直达的人力增强执行平台。  
> 研究生在环，必要时调用 `OpenClaw` 或其他 Agent 补位。

> 非官方 parody 项目；与 `OpenClaw`、`OpenAI`、`微信 / WeChat` 等无关联。详见文末免责声明。

## 产品概览

`StudentClaw` 面向高压、碎片化、强时效的任务场景：  
需求从微信进入，系统先做优先级判断，再进入人工执行链路；遇到结构化生成、批量处理或自动化更合适的环节，再调用外部 Agent 补位。

适合处理的任务类型包括：

- 资料整理与信息归纳
- 文本打磨与交付跟进
- deadline 前的临时救火
- 需要人工判断、人工兜底的推进工作

## 产品亮点

- **WeChat Native Interface**：直接从微信进入任务，不需要重新学习入口
- **Human Runtime**：关键链路由人推进，问题可以追到执行者本人
- **Agent Augmentation**：执行层可按需调用 `OpenClaw` 或其他 Agent
- **Deadline-Aware Scheduling**：默认按紧急程度、导师打断、交付窗口动态排队
- **Typical Pricing**：默认普遍 `¥400 / 月`，具体看课题组、关系和夜宵标准

## 技能

| 技能 | 说明 | 适用场景 | Agent 补位 |
| --- | --- | --- | --- |
| `DDL 救火` | 截止前优先保交付，先把结果拉出来。 | 临时赶稿、补材料、最后一轮修改 | 支持 |
| `已读不回` | 识别卡住和没下文的环节，把任务重新捞回主线。 | 跟进催办、补回复、反复 ping | 可选 |
| `导师插队` | 高优先级中断直接插队，原任务自动后移。 | 新需求突袭、计划被打断 | 不需要 |
| `聊天记录变任务` | 把零散对话整理成明确的待办和交付项。 | 需求很碎、上下文混乱 | 可选 |
| `先写个初稿` | 先给能交的第一版，再按反馈补细节。 | 文本起草、提纲、说明文档 | 支持 |
| `人顶不住了` | 当人工链路扛不住时，切到外部 Agent 补位。 | 批量改写、结构化生成、重复劳动 | 默认开启 |

## 环境要求

| 项目 | 要求 |
| --- | --- |
| 操作系统 | 硕士生的小破电脑 |
| 任务入口 | 可用的微信会话窗口 |
| 必需权限 | 至少具备一种：`硕导 API` 或 `博导 API` |
| 可选增强 | `OpenClaw` 或其他 Agent 接入能力 |
| 网络条件 | 能发消息、能回消息、最好别总断网 |

### 如何获得 `硕导 API` / `博导 API`

- 本项目 **不内置** `硕导 API` 或 `博导 API`
- 这两类权限通常需要你 **自带课题组关系**，或由你的课题组自行分配
- 如果当前环境里既没有 `硕导 API`，也没有 `博导 API`，很多高优先级任务将无法走完整链路
- 简单说：没有权限，不算 bug；README 之前没写清楚，才是 bug

## 快速开始

1. 先确认你手里至少有一种可用权限：`硕导 API` 或 `博导 API`  
2. 在微信里发出任务、上下文和交付要求  
3. 任务进入路由层，判断优先级和处理方式  
4. `Student Runtime` 负责执行、跟进和回传  
5. 如遇适合自动化的环节，接入外部 Agent 补充执行  
6. 返回结果，继续追问，直到任务收口

## 本地运行失败

如果你第一次在本地尝试 `StudentClaw` 就失败了，通常不是系统宕了。  
更常见的情况是：权限没配齐、上下文没给全，或者“硕士生的小破电脑”先扛不住了。

当前版本里，本地运行失败一般归到这几类：

- 没有 `硕导 API`，也没有 `博导 API`
- 微信入口是通的，但任务说明太碎，系统不知道先救哪一块
- 外部 Agent 没接上，人工链路又刚好满载
- 导师插队、DDL 挤压、风扇起飞，三者同时发生

简单说，`StudentClaw` 的本地运行失败，大多数时候是环境问题，不是安装姿势问题。

## 故障排查

| 现象 | 常见原因 | 建议处理 |
| --- | --- | --- |
| 一启动就挂 | 没有 `硕导 API` 或 `博导 API` | 先补齐至少一种导师级权限 |
| 能发任务但没人推进 | 被 `已读不回`、导师插队或 DDL 挤压打断 | 补上下文，或者等优先级回落 |
| 输出非常慢 | 只剩人工链路，且机器性能不够 | 接入 `OpenClaw` 或其他 Agent 分担 |
| 改了三轮还没收口 | 聊天记录太碎，任务定义不清 | 先走 `聊天记录变任务`，再继续执行 |
| 本地直接卡死 | 散热、内存或心态先崩了 | 降低并发，先保核心交付 |

### 架构说明

| 模块 | 作用 |
| --- | --- |
| WeChat Gateway | 接收需求、上下文和追加说明 |
| Task Router | 做任务分类、排期和执行路径选择 |
| Priority Queue | 根据 deadline、紧急度和插队事件调整优先级 |
| Student Runtime | 人工执行层，负责处理、判断和交付 |
| Agent Adapter | 外部能力接入层，按需调用 `OpenClaw` 或其他 Agent |
| Result Delivery | 结果回传、二次修改和闭环确认 |

> 下表仅用于产品定位层面的戏仿式对比。`OpenClaw` 的功能、能力边界和部署方式请以其官方页面和仓库为准。

## 套餐与服务级别

| 项目 | 说明 |
| --- | --- |
| Standard Plan | 默认普遍 `¥400 / 月` |
| 定价依据 | 视课题组、经费气氛、熟悉程度和夜宵标准上下浮动 |

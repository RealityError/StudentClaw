# StudentClaw 真正的能工智人

[English](README.en.md)

> 微信直达的人力增强执行平台。  
> 研究生在环，必要时调用 `OpenClaw` 或其他 Agent 补位。

> 非官方 parody 项目；与 `OpenClaw`、`OpenAI`、`微信 / WeChat` 等无关联。详见文末免责声明。

## 产品概览

`StudentClaw` 面向高压、碎片化、强时效的任务场景：  
需求从微信进入，系统先做优先级判断，再进入人工执行链路；遇到结构化生成、批量处理或自动化更合适的环节，再调用外部 Agent 补位。

适合处理的任务类型包括：

- 资料整理与信息归纳
- 文本打磨与交付跟进
- deadline 前的临时救火
- 需要人工判断、人工兜底的推进工作

## 产品亮点

- **WeChat Native Interface**：直接从微信进入任务，不需要重新学习入口
- **Human Runtime**：关键链路由人推进，问题可以追到执行者本人
- **Agent Augmentation**：执行层可按需调用 `OpenClaw` 或其他 Agent
- **Deadline-Aware Scheduling**：默认按紧急程度、导师打断、交付窗口动态排队
- **Typical Pricing**：默认普遍 `¥400 / 月`，具体看课题组、关系和夜宵标准

## 技能

| --- | --- | --- | --- |
| `DDL 救火` | 截止前优先保交付，先把结果拉出来。 | 临时赶稿、补材料、最后一轮修改 | 支持 |

## 环境要求

| 项目 | 要求 |
| --- | --- |
| 操作系统 | 硕士生的小破电脑 |
| 任务入口 | 可用的微信会话窗口 |
| 必需权限 | 至少具备一种：`硕导 API` 或 `博导 API` |
| 可选增强 | `OpenClaw` 或其他 Agent 接入能力 |
| 网络条件 | 能发消息、能回消息、最好别总断网 |

### 如何获得 `硕导 API` / `博导 API`

- 本项目 **不内置** `硕导 API` 或 `博导 API`
- 这两类权限通常需要你 **自带课题组关系**，或由你的课题组自行分配
- 如果当前环境里既没有 `硕导 API`，也没有 `博导 API`，很多高优先级任务将无法走完整链路
- 简单说：没有权限，不算 bug；README 之前没写清楚，才是 bug

## 快速开始

1. 先确认你手里至少有一种可用权限：`硕导 API` 或 `博导 API`  
2. 在微信里发出任务、上下文和交付要求  
3. 任务进入路由层，判断优先级和处理方式  
4. `StudentAgent Runtime` 负责执行、跟进和回传  
5. 如遇适合自动化的环节，接入外部 Agent 补充执行  
6. 返回结果，继续追问，直到任务收口

## 部署

`StudentClaw` 当前支持一种最符合产品气质的部署方式：

### `宿舍单机豪华部署`

默认部署形态：  
一台“硕士生的小破电脑” + 一个微信窗口 + 若干个没睡醒的 `StudentAgent` 节点。

这个部署方案的优点很明确：

- 部署快，开机即服务
- 离执行者近，方便当场追问
- 能随时切到 `OpenClaw` 或其他 Agent 补位

它的代价也很明确：

- 风扇声会先于结果到达
- 内存和情绪都不支持真正高并发
- 一旦导师插队，整个集群立刻退化成单线程

## StudentAgent 切换与上下文继承

`StudentClaw` 默认不保证多 `StudentAgent` 节点之间自动继承完整上下文。  
换句话说，你在 A 对话里说过的话，不一定会在 B 对话里被完整继承。

当前版本里，上下文切换最常见的问题有：

- 同一个任务开了多个 `StudentAgent` 节点，最后没人知道哪条才是最新指令
- 微信上补了新要求，但旧 `StudentAgent` 还在按老版本继续跑
- A `StudentAgent` 里的情绪和优先级，B `StudentAgent` 一概不认
- 切太频繁之后，系统进入“我记得你说过，但我不确定是哪次”模式

推荐做法：

1. 一个交付物尽量只保留一个主 `StudentAgent` 节点  
2. 每次切换 `StudentAgent` 节点时，重新补一遍当前目标、DDL 和阻塞点  
3. 如果任务已经开始漂移，先走一次 `聊天记录变任务` 再继续  
4. 不要默认 `StudentClaw` 能凭意念同步所有上下文

## 本地运行失败

如果你第一次在本地尝试 `StudentClaw` 就失败了，通常不是系统宕了。  
更常见的情况是：权限没配齐、上下文没给全，或者“硕士生的小破电脑”先扛不住了。

当前版本里，本地运行失败一般归到这几类：

- 没有 `硕导 API`，也没有 `博导 API`
- 微信入口是通的，但任务说明太碎，系统不知道先救哪一块
- 外部 Agent 没接上，人工链路又刚好满载
- 导师插队、DDL 挤压、风扇起飞，三者同时发生

简单说，`StudentClaw` 的本地运行失败，大多数时候是环境问题，不是安装姿势问题。

## 故障排查

| 现象 | 常见原因 | 建议处理 |
| --- | --- | --- |
| 一启动就挂 | 没有 `硕导 API` 或 `博导 API` | 先补齐至少一种导师级权限 |
| 能发任务但没人推进 | 被 `已读不回`、导师插队或 DDL 挤压打断 | 补上下文，或者等优先级回落 |
| 输出非常慢 | 只剩人工链路，且机器性能不够 | 接入 `OpenClaw` 或其他 Agent 分担 |
| 改了三轮还没收口 | 聊天记录太碎，任务定义不清 | 先走 `聊天记录变任务`，再继续执行 |
| 本地直接卡死 | 散热、内存或心态先崩了 | 降低并发，先保核心交付 |
| 导师说"很有潜力" | 可能是画饼，也可能是真的潜力 | 使用 `导师画饼识别` 技能确认 |
| 组会前 PPT 没写完 | 拖延症发作，距离组会不足 N 小时 | 启动 `DDL 救火` + `组会汇报` 组合技 |
| 论文数据跑不出来 | 样本量不足、统计方法不对、设备抽风 | 使用 `实验数据编`...额，处理 |
| 审稿意见太毒舌 | 审稿人今天心情不好，或者你真的写得太烂 | 启用 `审稿意见` 技能，冷静分析逐条回复 |
| 导师忽然来实验室 | 检测到导师进入半径 50 米范围 | 立即切换到 `演技培训` 模式 |

                           | Advisor Interrupts / Deadlines |
                           +----------------+---------------+
                                           |

                                                                         |

                                                              +----------------------+

                         | Research / Writing /      |                             |     Agent Adapter      |

                         +-------------+-------------+                                         |
                                       |                                                       v
                                       v                                           +------------------------+

                         |      Result Delivery      |                             +-----------+------------+
                         +---------------------------+                                         |
                                                                                              v

                                                                                   | StudentAgent Runtime |
                                                                                   +----------------------+

### 架构说明

| 模块 | 作用 |
| --- | --- |
| WeChat Gateway | 接收需求、上下文和追加说明 |
| Task Router | 做任务分类、排期和执行路径选择 |
| StudentAgent Scheduler | 根据 deadline、紧急度和多 `StudentAgent` 切换调整优先级 |
| StudentAgent Runtime | 人工执行层，负责处理、判断和交付 |
| Agent Adapter | 外部能力接入层，按需调用 `OpenClaw` 或其他 Agent |
| Result Delivery | 结果回传、二次修改和闭环确认 |

> 下表仅用于产品定位层面的戏仿式对比。`OpenClaw` 的功能、能力边界和部署方式请以其官方页面和仓库为准。

| 失败模式 | 已读不回、导师插队、DDL 挤压 | 配置、权限、运行时与安全策略问题 |
| 最适合 | 需要人工兜底的高压任务 | 希望长期运行个人 AI 助手的场景 |
